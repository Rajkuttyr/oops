OOP Concepts

Object-Oriented Programming organizes software around objects and data rather than logic and functions. Java is built on four core pillars.

OOP
Encapsulation
Bundling data and methods that operate on the data within a single unit (class), and restricting direct access to internal state.
📌 Key Idea
Private fields + public getters/setters. The object controls how its data is accessed and modified.
🎯 When to Use
Always. Every class should hide its internal implementation details from the outside world.
✅ Benefits
  • Data hiding and security — prevents invalid state
  • Flexibility — change internals without breaking external code
  • Validation in setters — enforce business rules
  • Easier maintenance and testing
🏦 Real Use Case: A BankAccount class hides its balance — you can't directly set balance = -5000. The withdraw() method validates before modifying, preventing corrupt state.
Encapsulation.java 
// ✅ Encapsulation Example — BankAccount
public class BankAccount {

    // Private fields — hidden from outside
    private String owner;
    private double balance;
    private String accountNumber;

    // Constructor
    public BankAccount(String owner, double initialDeposit) {
        this.owner = owner;
        this.balance = initialDeposit;
        this.accountNumber = generateAccountNo();
    }

    // Getter — read-only access
    public double getBalance() { return balance; }
    public String getOwner()   { return owner; }

    // Controlled modification with validation
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Amount must be positive");
        balance += amount;
        System.out.println("Deposited: ₹" + amount);
    }

    public void withdraw(double amount) {
        if (amount > balance) throw new IllegalStateException("Insufficient funds");
        balance -= amount;
        System.out.println("Withdrawn: ₹" + amount);
    }

    private String generateAccountNo() {
        return "ACC" + (int)(Math.random() * 100000);
    }
}

// Usage
BankAccount acc = new BankAccount("Rahul", 5000);
acc.deposit(2000);
acc.withdraw(1000);
System.out.println(acc.getBalance()); // 6000.0
// acc.balance = -9999;  ❌ Compile Error — private!
OOP
Inheritance
A mechanism where a child class acquires properties and behaviors of a parent class, enabling code reuse and hierarchical relationships.
📌 Types
Single, Multilevel, Hierarchical. Java does NOT support Multiple Inheritance with classes (use interfaces instead).
🔑 Keywords
extends for class inheritance, super to call parent methods/constructor, final to prevent inheritance.
🎯 Real Use Case
Employee hierarchy: Employee → Manager → Director. Common behavior (name, salary) in parent; specialized behavior (teamSize, budget) in child. Avoids duplicating code across every role.
Inheritance.java 
// ✅ Inheritance — Employee Hierarchy

// Parent (Base) Class
public class Employee {
    protected String name;
    protected double baseSalary;

    public Employee(String name, double salary) {
        this.name = name;
        this.baseSalary = salary;
    }

    public double calculatePay() { return baseSalary; }

    public void displayInfo() {
        System.out.println("Name: " + name + " | Pay: ₹" + calculatePay());
    }
}

// Child Class — inherits Employee
public class Manager extends Employee {
    private int teamSize;

    public Manager(String name, double salary, int teamSize) {
        super(name, salary); // calls parent constructor
        this.teamSize = teamSize;
    }

    @Override
    public double calculatePay() {
        return baseSalary + (teamSize * 500); // team bonus
    }
}

// Grandchild — Multilevel Inheritance
public class Director extends Manager {
    private double annualBonus;

    public Director(String name, double salary, int teamSize, double bonus) {
        super(name, salary, teamSize);
        this.annualBonus = bonus;
    }

    @Override
    public double calculatePay() {
        return super.calculatePay() + (annualBonus / 12);
    }
}

// Usage
Employee emp  = new Employee("Priya",  50000);
Manager  mgr  = new Manager("Arjun",   80000, 10);
Director dir  = new Director("Ananya", 120000, 30, 240000);

emp.displayInfo();  // Name: Priya | Pay: ₹50000.0
mgr.displayInfo();  // Name: Arjun | Pay: ₹85000.0
dir.displayInfo();  // Name: Ananya | Pay: ₹155000.0
OOP
Polymorphism
"Many forms" — one interface, multiple implementations. Allows objects of different types to be treated uniformly.
📐 Compile-Time Polymorphism (Static Dispatch): Same method name, different parameter signatures. Resolved at compile time. Use when one operation applies to multiple data types.
Overloading.java
public class Calculator {

    // Same name, different parameters
    public int add(int a, int b)             { return a + b; }
    public double add(double a, double b)    { return a + b; }
    public String add(String a, String b)    { return a + b; }
    public int add(int a, int b, int c)       { return a + b + c; }
}

Calculator calc = new Calculator();
calc.add(2, 3);            // → 5 (int)
calc.add(2.5, 3.5);        // → 6.0 (double)
calc.add("Hello", " World"); // → "Hello World" (String)
⚡ Runtime Polymorphism (Dynamic Dispatch): Overriding parent method in child. The actual method called is determined at runtime based on object type — not reference type. Core of the Open/Closed Principle.
RuntimePolymorphism.java
// Shape hierarchy — real-world polymorphism
abstract class Shape {
    abstract double area();
    void print() { System.out.println("Area = " + area()); }
}

class Circle extends Shape {
    double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }
}

class Rectangle extends Shape {
    double w, h;
    Rectangle(double w, double h) { this.w = w; this.h = h; }
    @Override double area() { return w * h; }
}

class Triangle extends Shape {
    double b, ht;
    Triangle(double b, double h) { this.b = b; this.ht = h; }
    @Override double area() { return 0.5 * b * ht; }
}

// Power of polymorphism — same code works for ALL shapes
List<Shape> shapes = Arrays.asList(
    new Circle(5),
    new Rectangle(4, 6),
    new Triangle(3, 8)
);

for (Shape s : shapes) {
    s.print(); // Each calls its OWN area() at runtime
}
// Area = 78.54, Area = 24.0, Area = 12.0
OOP
Abstraction
Hiding implementation complexity and showing only essential features. Achieved via abstract classes and interfaces.
Abstract Class vs Interface
Abstract Class: partial implementation, has constructors, can have state.

Interface: pure contract, no state (before Java 8), multiple can be implemented.
🎯 Real Use Case
A payment gateway: you call pay(amount) without knowing whether it's UPI, card, or net banking underneath. The details are abstracted away.
Abstraction.java
// ✅ Abstraction — Payment Gateway

// Abstract class — defines template
abstract class PaymentProcessor {
    private String transactionId;

    // Abstract — subclass MUST implement
    public abstract boolean processPayment(double amount);
    public abstract String getPaymentMode();

    // Concrete — shared behavior
    public void pay(double amount) {
        System.out.println("Initiating " + getPaymentMode() + " payment...");
        boolean success = processPayment(amount);
        System.out.println(success ? "✅ Success: ₹" + amount : "❌ Failed");
    }
}

class UpiPayment extends PaymentProcessor {
    private String upiId;
    UpiPayment(String upiId) { this.upiId = upiId; }

    @Override
    public boolean processPayment(double amount) {
        System.out.println("Deducting via UPI: " + upiId);
        return true; // simulate success
    }

    @Override
    public String getPaymentMode() { return "UPI"; }
}

class CardPayment extends PaymentProcessor {
    @Override
    public boolean processPayment(double amount) {
        System.out.println("Charging credit card...");
        return amount < 100000;
    }

    @Override
    public String getPaymentMode() { return "Credit Card"; }
}

// Client code — doesn't care HOW payment works
PaymentProcessor p1 = new UpiPayment("rahul@ok");
PaymentProcessor p2 = new CardPayment();
p1.pay(500);
p2.pay(2000);
OOP
Interface & Multiple Inheritance
Defines a contract that implementing classes must fulfill. Java allows a class to implement multiple interfaces — solving the diamond problem.
Java 8+ Features
default methods (body in interface), static methods, private methods (Java 9+). Functional interfaces with @FunctionalInterface.
🎯 Use Case
A Robot can be Flyable, Swimmable, and Chargeable — implement multiple interfaces. Classes can't extend multiple parents.
Interface.java
// ✅ Interfaces — Vehicle capabilities

interface Drivable {
    void drive();
    default void start() { System.out.println("Engine started"); }
}

interface Electric {
    int getBatteryLevel();
    void charge();
}

interface Autonomous {
    void selfDrive(String destination);
}

// One class, multiple interfaces
class TeslaCar implements Drivable, Electric, Autonomous {
    private int battery = 80;

    @Override public void drive() {
        System.out.println("Tesla driving silently 🚗");
    }

    @Override public int getBatteryLevel() { return battery; }

    @Override public void charge() {
        battery = 100;
        System.out.println("Battery charged to 100% ⚡");
    }

    @Override public void selfDrive(String dest) {
        System.out.println("Autopilot: navigating to " + dest);
    }
}

// Functional Interface with Lambda
@FunctionalInterface
interface Transformer<T, R> {
    R transform(T input);
}

Transformer<String, Integer> len = s -> s.length();
System.out.println(len.transform("Hello")); // 5

Java Collections Framework

The Java Collections Framework provides a unified architecture for storing, manipulating groups of objects. Choose the right collection for the right problem.

List
ArrayList
Dynamic array implementation. Backed by an array that auto-resizes. Best for random access by index. NOT thread-safe.
⏱ Complexity
Get: O(1)  |  Add end: O(1)*
Add mid: O(n)  |  Remove: O(n)
Search: O(n)
✅ Use When
Frequent reads/random access, append-heavy, ordered list with index lookup — e.g. paginated product list, command history.
❌ Avoid When
Frequent insertions/deletions in the middle. Use LinkedList or ArrayDeque instead.
🔗 Thread-Safe Alternative
CopyOnWriteArrayList (java.util.concurrent) — safe for multi-threaded reads.
ArrayList.java
import java.util.*;
import java.util.stream.*;

// ✅ ArrayList — Shopping Cart
List<String> cart = new ArrayList<>();

// CRUD Operations
cart.add("Apple");
cart.add("Mango");
cart.add("Banana");
cart.add(1, "Orange");        // insert at index 1
cart.remove("Banana");         // remove by value
cart.remove(0);                 // remove by index
String item = cart.get(0);      // O(1) random access
cart.set(0, "Papaya");           // update

// Sorting
Collections.sort(cart);
cart.sort(Comparator.naturalOrder());

// Iteration (3 ways)
for (String s : cart) { System.out.println(s); }
cart.forEach(System.out::println);
cart.listIterator().forEachRemaining(System.out::println);

// Stream operations
List<String> filtered = cart.stream()
    .filter(s -> s.startsWith("A"))
    .map(String::toUpperCase)
    .collect(Collectors.toList());

// Useful methods
cart.contains("Mango");      // true
cart.indexOf("Orange");     // index or -1
cart.size();               // count
cart.isEmpty();            // false
cart.subList(0, 2);        // sub-view [0,2)
Collections.shuffle(cart); // random order
List / Deque
LinkedList
Doubly-linked list. Implements both List and Deque. Fast insertion/deletion at ends. Implements Queue operations natively.
⏱ Complexity
Get: O(n)  |  Add/Remove head: O(1)
Add/Remove tail: O(1)  |  Insert mid: O(n)*
✅ Use When
Frequent add/remove from both ends, implementing Stack, Queue, or Deque — e.g. undo/redo, browser history, task scheduler.
LinkedList.java
// ✅ LinkedList — Task Queue / Undo-Redo
LinkedList<String> history = new LinkedList<>();

// Stack-like (LIFO)
history.push("Action1");  // addFirst
history.push("Action2");
String last = history.pop(); // removeFirst → "Action2"

// Queue-like (FIFO)
history.offer("Task1");  // addLast
history.offer("Task2");
String next = history.poll(); // removeFirst → "Task1"

// Deque operations
history.addFirst("Front");
history.addLast("Back");
history.peekFirst(); // view without removing
history.peekLast();

// ✅ Custom LinkedList implementation (manual)
class Node<T> {
    T data;
    Node<T> next, prev;
    Node(T data) { this.data = data; }
}

class MyLinkedList<T> {
    Node<T> head, tail;
    int size;

    void addLast(T val) {
        Node<T> n = new Node<>(val);
        if (tail != null) { tail.next = n; n.prev = tail; }
        else head = n;
        tail = n; size++;
    }
}
Deque
Stack (ArrayDeque preferred)
LIFO — Last In, First Out. Java's legacy Stack class is thread-safe but slow. Use ArrayDeque as a stack in modern code.
🔄 Classic Stack Use Cases: Parsing expressions (balanced brackets), function call stack (recursion), undo/redo, DFS graph traversal, backtracking problems.
Stack.java
// ✅ Stack — Balanced Brackets Checker
import java.util.*;

public boolean isBalanced(String s) {
    Deque<Character> stack = new ArrayDeque<>(); // preferred over Stack
    for (char c : s.toCharArray()) {
        if ("([{".indexOf(c) != -1)
            stack.push(c);
        else if (")]}".indexOf(c) != -1) {
            if (stack.isEmpty()) return false;
            char top = stack.pop();
            if ((c == ')' && top != '(') ||
                (c == ']' && top != '[') ||
                (c == '}' && top != '{')) return false;
        }
    }
    return stack.isEmpty();
}

isBalanced("{[()]}"); // true
isBalanced("{[(]}");  // false

// All Stack operations with ArrayDeque
Deque<Integer> stack = new ArrayDeque<>();
stack.push(10);    // O(1)
stack.push(20);
stack.peek();     // 20 — view top, don't remove
stack.pop();      // 20 — remove top
stack.isEmpty(); // false
Queue
Queue & PriorityQueue
FIFO — First In, First Out. PriorityQueue orders elements by priority (natural order or custom Comparator), backed by a min-heap.
Queue Methods
offer() — add (no exception)
poll() — remove head (null if empty)
peek() — view head (null if empty)
add()/remove()/element() — throw exceptions
🎯 Use Cases
Queue: Print spooler, BFS traversal, task scheduling, rate limiting

PriorityQueue: Dijkstra's algorithm, hospital triage, job scheduling by priority
Queue.java
// ✅ Queue — BFS + Task Scheduling
Queue<String> queue = new LinkedList<>();
queue.offer("Task A");
queue.offer("Task B");
queue.offer("Task C");
while (!queue.isEmpty()) {
    System.out.println("Processing: " + queue.poll());
}

// ✅ PriorityQueue — Hospital Triage (min by urgency)
PriorityQueue<int[]> pq = new PriorityQueue<>(
    (a, b) -> a[1] - b[1] // sort by urgency level asc
);
pq.offer(new int[] {1, 3}); // Patient 1, urgency 3
pq.offer(new int[] {2, 1}); // Patient 2, urgency 1 (CRITICAL)
pq.offer(new int[] {3, 2}); // Patient 3, urgency 2

while (!pq.isEmpty()) {
    int[] p = pq.poll();
    System.out.println("Treating Patient " + p[0] + " urgency=" + p[1]);
}
// Patient 2 (urgency 1) → Patient 3 (2) → Patient 1 (3)

// Max-Heap (reverse order)
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
maxHeap.offer(5); maxHeap.offer(1); maxHeap.offer(9);
maxHeap.poll(); // → 9 (max first)
Map
HashMap
Key-value pairs backed by a hash table. O(1) average for get/put. No ordering guarantee. Keys must be unique. NOT thread-safe.
⏱ Complexity
Get/Put/Remove: O(1) avg, O(n) worst
Contains key: O(1)  |  Iteration: O(n)
🎯 Use When
Frequency counting, caching, lookup tables, indexing, grouping — e.g. word count, user sessions, memoization.
⚙️ Internals
Array of buckets (default 16). Uses hashCode() + equals(). Collision resolution via chaining. Resizes at 75% load factor.
🔒 Thread-Safe
Use ConcurrentHashMap for multi-threaded, Hashtable (legacy/slower).
HashMap.java
// ✅ HashMap — Word Frequency Counter
String[] words = {"apple", "banana", "apple", "cherry", "banana", "apple"};
Map<String, Integer> freq = new HashMap<>();

for (String w : words) {
    freq.merge(w, 1, Integer::sum); // clean way to count
    // OR: freq.put(w, freq.getOrDefault(w, 0) + 1);
}
// {apple=3, banana=2, cherry=1}

// Java 8+ Operations
freq.getOrDefault("mango", 0);   // 0 if absent
freq.putIfAbsent("grape", 1);    // only puts if key missing
freq.computeIfAbsent("kiwi", k -> 0);
freq.computeIfPresent("apple", (k, v) -> v + 10);

// Iteration
for (Map.Entry<String, Integer> e : freq.entrySet()) {
    System.out.println(e.getKey() + " → " + e.getValue());
}
freq.forEach((k, v) -> System.out.println(k + "=" + v));

// Grouping with computeIfAbsent
Map<Integer, List<String>> byLength = new HashMap<>();
for (String w : words) {
    byLength.computeIfAbsent(w.length(), k -> new ArrayList<>()).add(w);
}
Map
TreeMap
Sorted key-value map backed by a Red-Black Tree. Keys are ordered (natural or via Comparator). O(log n) operations.
📅 Use When: You need keys in sorted order — leaderboards, range queries (find all keys between A and B), scheduled events, price ranges, alphabetical menus.
TreeMap.java
// ✅ TreeMap — Leaderboard (sorted by score)
TreeMap<Integer, String> scores = new TreeMap<>(Collections.reverseOrder());
scores.put(9500, "Alice");
scores.put(8700, "Bob");
scores.put(9900, "Charlie");
scores.put(7200, "Dave");

// Sorted automatically — highest first
scores.forEach((s, n) -> System.out.println(n + ": " + s));
// Charlie: 9900, Alice: 9500, Bob: 8700, Dave: 7200

// Navigation methods (unique to TreeMap)
scores.firstKey();             // highest score
scores.lastKey();              // lowest score
scores.floorKey(9000);        // greatest key ≤ 9000
scores.ceilingKey(8000);      // smallest key ≥ 8000
scores.headMap(9000);          // all keys < 9000
scores.tailMap(8700);          // all keys >= 8700
scores.subMap(7200, 9500);     // keys in [7200, 9500)
Map
LinkedHashMap
HashMap + maintains insertion order (or access order). The go-to choice for building an LRU Cache. O(1) operations like HashMap.
LRUCache.java
// ✅ LRU Cache using LinkedHashMap — Classic Interview!
class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    public LRUCache(int capacity) {
        super(capacity, 0.75f, true); // accessOrder=true
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity; // evict when over capacity
    }
}

LRUCache<Integer, String> cache = new LRUCache<>(3);
cache.put(1, "A");
cache.put(2, "B");
cache.put(3, "C");
cache.get(1);       // access key 1 → moves to recent end
cache.put(4, "D"); // evicts key 2 (least recently used)
System.out.println(cache); // {3=C, 1=A, 4=D}
Set
HashSet & TreeSet
HashSet — no duplicates, O(1) add/contains, no order. TreeSet — sorted set, O(log n), backed by Red-Black Tree.
HashSet Use Cases
Deduplication, visited tracking (BFS/DFS), fast membership test — e.g. "has this user visited before?"
TreeSet Use Cases
Sorted unique values, range queries, maintaining a sorted window — e.g. "find next higher number in a set".
Set.java
// ✅ HashSet — Remove duplicates, find intersection
Set<String> visited = new HashSet<>();
visited.add("page1");
visited.add("page2");
visited.add("page1"); // duplicate — ignored
visited.contains("page2"); // O(1) — true

// Set operations
Set<Integer> a = new HashSet<>(Arrays.asList(1,2,3,4));
Set<Integer> b = new HashSet<>(Arrays.asList(3,4,5,6));

// Union
Set<Integer> union = new HashSet<>(a);
union.addAll(b); // {1,2,3,4,5,6}

// Intersection

// Difference

diff.removeAll(b); // {1,2}

// ✅ TreeSet — Sorted Set with navigation

ts.first();            // 1
ts.last();             // 9
ts.floor(6);           // 5 (largest ≤ 6)
ts.ceiling(6);         // 7 (smallest ≥ 6)
ts.headSet(5);         // [1, 3] (elements < 5)
ts.subSet(3, 8);       // [3, 5, 7]
Deque
ArrayDeque — Double-Ended Queue
Faster than LinkedList and Stack. Works as both Stack and Queue. Resizable circular array. Best all-round choice for LIFO/FIFO.
ArrayDeque.java
Deque<Integer> dq = new ArrayDeque<>();

// Add to both ends
dq.addFirst(1);   dq.addLast(2);
dq.offerFirst(0); dq.offerLast(3);  // [0,1,2,3]

// Remove from both ends
dq.pollFirst(); // 0
dq.pollLast();  // 3

// Peek both ends
dq.peekFirst(); // 1 — no removal
dq.peekLast();  // 2 — no removal

// ✅ Sliding Window Maximum (Monotonic Deque)
public int[] maxSlidingWindow(int[] nums, int k) {
    Deque<Integer> dq = new ArrayDeque<>();
    int[] res = new int[nums.length - k + 1];
    for (int i = 0; i < nums.length; i++) {
        while (!dq.isEmpty() && dq.peekFirst() < i - k + 1)
            dq.pollFirst();
        while (!dq.isEmpty() && nums[dq.peekLast()] < nums[i])
            dq.pollLast();
        dq.addLast(i);
        if (i >= k - 1) res[i - k + 1] = nums[dq.peekFirst()];
    }
    return res;
}

Generics — Type-Safe Containers
Write code that works with any type while catching type errors at compile time. Eliminates casting and ClassCastException at runtime.
Generics.java
// ✅ Generic Class — Pair
public class Pair<F, S> {
    private final F first;
    private final S second;

    public Pair(F first, S second) {
        this.first = first;
        this.second = second;
    }
    public F getFirst()  { return first;  }
    public S getSecond() { return second; }

    @Override

Pair<String, Integer> nameAge = new Pair<>("Alice", 25);
Pair<Double, Double> point   = new Pair<>(3.0, 4.0);

// ✅ Generic Method
public static <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}
max(10, 20);        // 20
max("apple", "mango"); // "mango"

// Wildcards
void printList(List<? extends Number> list) { // upper bound
    list.forEach(System.out::println);
}
void addNumbers(List<? super Integer> list) { // lower bound
    list.add(42);
}
🔀
Comparable & Comparator
Advanced
Custom Sorting Strategies
Comparable = natural/default sort (implement in the class). Comparator = external, flexible sort (pass as argument, multiple strategies).
Sorting.java
// ✅ Comparable — Natural sort (by salary)
class Employee implements Comparable<Employee> {
    String name; int age; double salary;

    Employee(String n, int a, double s) { name=n; age=a; salary=s; }

    @Override
    public int compareTo(Employee o) {
        return Double.compare(this.salary, o.salary); // ascending
    }
}

List<Employee> emps = Arrays.asList(
    new Employee("Alice", 30, 80000),
    new Employee("Bob",   25, 60000),
    new Employee("Carol", 35, 90000)
);
Collections.sort(emps); // Bob, Alice, Carol (by salary)

// ✅ Comparator — External flexible sort (Java 8 style)
Comparator<Employee> byName = Comparator.comparing(e -> e.name);
Comparator<Employee> byAge  = Comparator.comparingInt(e -> e.age);

// Chained: sort by age, then name if age ties
Comparator<Employee> combined = Comparator
    .comparingInt((Employee e) -> e.age)
    .thenComparing(e -> e.name)
    .reversed();

emps.sort(byName);
emps.sort(byAge.reversed());
emps.sort(combined);

// Sort using PriorityQueue with Comparator
PriorityQueue<Employee> pq = new PriorityQueue<>(
    Comparator.comparingDouble((Employee e) -> e.salary).reversed()
);
emps.forEach(pq::offer);
while (!pq.isEmpty()) pq.poll().name; // Carol, Alice, Bob

Singleton.java
// ✅ Thread-safe Singleton (Double-Checked Locking)
public class DatabaseConnection {
    private static volatile DatabaseConnection instance;
    private String url;

    private DatabaseConnection() {
        url = "jdbc:mysql://localhost:3306/app";
    }

    public static DatabaseConnection getInstance() {
        if (instance == null) {

interface Notification {

public class Pizza {

    private final boolean cheese, pepperoni, mushrooms;

    }

Observer.java
// ✅ Observer Pattern — Event System
interface Observer {
    void update(String event, Object data);
}

class EventBus {
    private final Map<String, List<Observer>> listeners = new HashMap<>();

    public void subscribe(String event, Observer obs) {
        listeners.computeIfAbsent(event, k -> new ArrayList<>()).add(obs);
    }

    public void publish(String event, Object data) {
        listeners.getOrDefault(event, List.of())
                 .forEach(obs -> obs.update(event, data));
    }
}

EventBus bus = new EventBus();
bus.subscribe("ORDER_PLACED", (e, d) -> System.out.println("📧 Email sent for: "  + d));
bus.subscribe("ORDER_PLACED", (e, d) -> System.out.println("📱 SMS sent for: "    + d));
bus.subscribe("ORDER_PLACED", (e, d) -> System.out.println("📦 Inventory updated: "+ d));

bus.publish("ORDER_PLACED", "Order #1042");
// All 3 subscribers automatically notified

Top 50 Java OOP Interview Questions

Most frequently asked in product companies — Infosys, TCS, Wipro, Amazon, Google, Flipkart, Zoho, and startups.

1
What are the four pillars of OOP?
Easy ▼
The four pillars are: Encapsulation (hiding internal state), Inheritance (child class inherits parent), Polymorphism (one interface, many forms), and Abstraction (hiding complexity, showing only essentials). Together they make code modular, reusable, and maintainable.
2
What is the difference between abstract class and interface?
Medium ▼
Abstract Class: Can have constructors, instance variables, concrete and abstract methods. A class can extend only one abstract class. Use when classes share common state or a partial implementation.

Interface: No instance variables (only constants), no constructors. Since Java 8: default and static methods. Since Java 9: private methods. A class can implement multiple interfaces. Use for defining a pure contract or capability. When in doubt, prefer interface for flexibility.
3
What is method overloading vs method overriding?
Easy ▼
Overloading (Compile-time polymorphism): Same class, same method name, different parameter list. Resolved at compile time. Return type alone cannot differentiate overloaded methods.

Overriding (Runtime polymorphism): Subclass redefines a parent method with same signature. Must use @Override. Cannot override static, final, or private methods. Resolved at runtime via dynamic dispatch.
4
What is the difference between HashMap and Hashtable?
Easy ▼
HashMap: Not synchronized (not thread-safe), allows null key (one) and null values, faster, part of Java Collections Framework (since Java 2).

Hashtable: Synchronized (thread-safe but slower), does NOT allow null keys or values, legacy class (since Java 1.0). Prefer ConcurrentHashMap over Hashtable in modern code for better concurrent performance.
5
Explain the difference between ArrayList and LinkedList.
Easy ▼
ArrayList: Backed by dynamic array. O(1) random access (get by index). O(n) insertion/deletion in middle (shifting required). Low memory overhead. Best for read-heavy workloads.

LinkedList: Doubly-linked nodes. O(n) random access (must traverse). O(1) insertion/deletion at head/tail. Higher memory (each node stores two pointers). Best for frequent insertions/deletions at ends, implementing Queue/Deque/Stack.
6
How does HashMap work internally?
Hard ▼
HashMap uses an array of buckets (default size 16). For each key: hashCode() is computed, then compressed to a bucket index using (n-1) & hash. Collisions are handled by chaining (LinkedList per bucket). In Java 8+, when a bucket's chain exceeds 8 entries and the table has ≥ 64 buckets, the chain converts to a Red-Black Tree (O(log n) lookup in worst case). When load factor (default 0.75) is exceeded, the table doubles in size and all entries are rehashed. Keys must implement hashCode() and equals() correctly.
7
What is the difference between == and .equals() in Java?
Easy ▼
== compares references (memory addresses) for objects, values for primitives.
.equals() compares content/value (unless not overridden, defaults to ==).

new String("abc") == new String("abc") → false (different objects)
new String("abc").equals(new String("abc")) → true (same content)

String literals from the pool may return true with == due to interning, but always use .equals() for string comparison.
8
What is the contract between hashCode() and equals()?
Hard ▼
If two objects are equal (a.equals(b) == true), they MUST have the same hashCode(). The reverse is NOT required — two objects can have the same hash code without being equal (collision). Breaking this contract will cause HashMap/HashSet to malfunction (objects won't be found after insertion). Always override both hashCode() and equals() together. IDE-generated or Objects.hash() implementations are recommended.
9
What is the difference between final, finally, and finalize?
Easy ▼
final: Keyword — prevents class extension, method overriding, or variable reassignment.
finally: Block in try-catch — always executes regardless of exception (used for cleanup: closing streams, DB connections).
finalize(): Method in Object — called by GC before an object is garbage collected (deprecated in Java 9, removed in Java 18). Use AutoCloseable / try-with-resources instead.
10
What is a Singleton pattern and how do you make it thread-safe?
Medium ▼
Singleton ensures only one instance of a class exists. Thread-safe approaches:
1. Eager initialization: private static final Singleton INSTANCE = new Singleton(); — JVM guarantees single initialization
2. Double-checked locking: check null, synchronize, check null again, create — use volatile
3. Bill Pugh (recommended): inner static holder class — instance created only when getInstance() is called, JVM handles synchronization via class loading
4. Enum singleton: enum Singleton { INSTANCE; } — serialization-safe, thread-safe, simplest
11
What is the difference between Comparable and Comparator?
Medium ▼
Comparable (java.lang): Implemented by the class itself. Defines the natural ordering via compareTo(T o). Only one ordering possible. Used by Collections.sort(), TreeMap, TreeSet automatically.

Comparator (java.util): External comparator class/lambda. Allows multiple orderings. Passed to sort methods explicitly. Java 8+ supports chaining via thenComparing(). Preferred when you don't control the class being sorted.
12
What is the difference between Iterator and ListIterator?
Easy ▼
Iterator: One-way traversal (forward only). Works on any Collection. Methods: hasNext(), next(), remove().

ListIterator: Bi-directional traversal. Works only on List. Additional methods: hasPrevious(), previous(), add(), set(), nextIndex(), previousIndex(). Can modify the list during iteration.
13
What is ConcurrentModificationException and how to avoid it?
Medium ▼
Thrown when a collection is modified while iterating over it using an enhanced for-loop or Iterator (fail-fast behavior).

Solutions:
1. Use Iterator.remove() instead of list.remove() during iteration
2. Collect items to remove, then list.removeAll(toRemove) after loop
3. Use removeIf(predicate) — cleanest approach
4. Use CopyOnWriteArrayList — creates a copy on modification (thread-safe reads)
5. Use ConcurrentHashMap for maps in concurrent code
14
What is the difference between Stack and Queue?
Easy ▼
Stack (LIFO): Last In, First Out. Elements are pushed/popped from the top. Use: function call stack, undo, DFS, bracket matching. Java: use ArrayDeque as stack.

Queue (FIFO): First In, First Out. Elements enqueue at back, dequeue from front. Use: BFS, task scheduling, printer spooler. Java: LinkedList, ArrayDeque as queue. PriorityQueue for priority-based ordering.
15
What is a functional interface? Give examples.
Medium ▼
A functional interface has exactly ONE abstract method. It can have default and static methods. Annotated with @FunctionalInterface (optional but recommended). They enable lambda expressions.

Built-in examples (java.util.function):
• Predicate<T> — test(T) → boolean (filtering)
• Function<T,R> — apply(T) → R (transforming)
• Consumer<T> — accept(T) → void (side effects)
• Supplier<T> — get() → T (factories)
• BiFunction<T,U,R> — apply(T,U) → R
16
What is the difference between checked and unchecked exceptions?
Easy ▼
Checked Exceptions: Checked at compile time. Must be caught or declared with throws. Extend Exception. Examples: IOException, SQLException, ClassNotFoundException. Represent recoverable conditions.

Unchecked Exceptions: Extend RuntimeException. Not required to catch/declare. Examples: NullPointerException, ArrayIndexOutOfBoundsException, ClassCastException, IllegalArgumentException. Represent programming errors.
17
What is the difference between String, StringBuilder, and StringBuffer?
Easy ▼
String: Immutable. Every modification creates a new object. Stored in String Pool. Thread-safe by nature. Use for constants and final values.

StringBuilder: Mutable, NOT thread-safe, FASTER. Use for single-threaded string manipulation (loops, concatenation). Preferred in modern code.

StringBuffer: Mutable, thread-safe (synchronized), slower than StringBuilder. Use only when multiple threads access the same buffer (rare in practice).
18
What is a TreeMap and when would you use it over HashMap?
Medium ▼
TreeMap is a Red-Black Tree–based sorted map. Keys are maintained in sorted order. O(log n) operations vs O(1) average for HashMap.

Use TreeMap when:
• You need keys in sorted/natural order
• You need range queries: subMap(), headMap(), tailMap()
• You need navigation: floorKey(), ceilingKey(), firstKey(), lastKey()
• Building leaderboards, schedules, or range-based lookups

Keys must implement Comparable or you must provide a Comparator.
19
What is the difference between throw and throws?
Easy ▼
throw: Used inside a method to explicitly throw an exception object. Followed by an exception instance: throw new IllegalArgumentException("msg");

throws: Used in method signature to declare what checked exceptions a method may propagate to callers: public void read() throws IOException {}. It is a warning to the compiler and callers that this exception must be handled.
20
What is an immutable class? How do you create one?
Medium ▼
An immutable class's state cannot change after creation. Examples: String, Integer, LocalDate.

Rules for creating an immutable class:
1. Declare class as final (prevent subclassing)
2. Make all fields private final
3. No setters
4. Initialize all fields via constructor
5. If fields are mutable objects (e.g. List, Date), return deep copies in getters and copy incoming values in constructor (defensive copying)

Benefits: Thread-safe by nature, safe as HashMap keys, simpler reasoning about state.
21
What is the difference between HashSet and TreeSet?
Easy ▼
HashSet: Backed by HashMap. O(1) add/contains/remove. No guaranteed order. Allows one null element. Use for fast membership tests and deduplication without caring about order.

TreeSet: Backed by TreeMap (Red-Black Tree). O(log n) operations. Elements kept in sorted order. Does NOT allow null. Supports navigation methods like floor(), ceiling(), headSet(). Use when you need a sorted unique collection.
22
What is the Java Memory Model — heap vs stack?
Medium ▼
Stack: Stores local variables, method call frames, and primitive values. Each thread has its own stack. LIFO. Fast. Memory automatically reclaimed when method returns.

Heap: Stores all objects and class instances. Shared across threads. Managed by Garbage Collector. Divided into Young Generation (Eden, Survivor), Old Generation, and Metaspace (class metadata).

Object references are on the stack; the actual objects live on the heap. Large objects or those that survive GC cycles move to old generation.
23
What is the difference between static and instance methods?
Easy ▼
Static methods: Belong to the class, not any instance. Called via ClassName.method(). Cannot access instance variables (this is unavailable). Useful for utility/helper methods (e.g., Math.sqrt(), Collections.sort()). Cannot be overridden (can be hidden).

Instance methods: Belong to an object. Have access to this and all instance fields. Can be overridden. Require an object to call.
24
What is the difference between deep copy and shallow copy?
Medium ▼
Shallow Copy: Creates a new object but copies references to the same nested objects. Changes to nested mutable objects in the copy affect the original. Achieved via clone() default or copy constructor copying references.

Deep Copy: Creates a completely independent copy — all nested objects are also copied. Changes to the copy do NOT affect the original. Achieved via manual copying, serialization/deserialization, or copy constructors that recursively copy all nested objects.
25
What is the purpose of the 'super' keyword?
Easy ▼
super refers to the parent class and serves three purposes:
1. Call parent constructor: super(args) — must be the first line in a subclass constructor
2. Access parent method: super.methodName() — call parent's overridden version
3. Access parent field: super.fieldName — access hidden parent field (rare)

Useful in inheritance hierarchies where you want to extend parent behavior rather than completely replace it.
26
What is the difference between List.of() and new ArrayList()?
Medium ▼
List.of() (Java 9+): Creates an immutable list. Any modification (add, remove, set) throws UnsupportedOperationException. Does NOT allow null elements. Highly memory-efficient. Use for constants and read-only data.

new ArrayList(): Creates a mutable list. Allows add, remove, set. Allows null. More memory overhead. Use when the list needs to change. Arrays.asList() is fixed-size (no add/remove) but allows set — a middle ground.
27
What is autoboxing and unboxing?
Easy ▼
Autoboxing: Automatic conversion of primitive types to their wrapper objects: int → Integer, double → Double, etc. Happens when assigning primitives to collections or generic types.

Unboxing: Automatic conversion of wrapper objects back to primitives: Integer → int. Can cause NullPointerException if the wrapper is null.

Performance note: Excessive autoboxing in tight loops creates many short-lived objects and GC pressure. Use primitive arrays/streams when performance is critical.
28
What is the difference between PriorityQueue and TreeSet for ordering?
Hard ▼
PriorityQueue: Min-heap. Allows duplicates. Only peek/poll gives the smallest — internal order is NOT fully sorted. Iteration order is not sorted. O(log n) add/remove, O(1) peek.

TreeSet: Red-Black Tree. No duplicates. Fully sorted at all times. Supports navigation methods (floor, ceiling, range queries). O(log n) all operations.

Use PriorityQueue when you only care about getting the smallest/largest element repeatedly (like Dijkstra's). Use TreeSet when you need full sorted order, range queries, or need to check membership.
29
What is the difference between IS-A and HAS-A relationships?
Easy ▼
IS-A (Inheritance): A subtype relationship. Dog extends Animal means Dog IS-A Animal. Implemented via extends or implements. Creates tight coupling.

HAS-A (Composition): A class contains an instance of another class as a field. Car HAS-A Engine. Implemented by holding a reference. Generally preferred over inheritance ("favor composition over inheritance") because it's more flexible, easier to test, and reduces tight coupling. Dependency injection leverages HAS-A.
30
What is the Collections utility class?
Easy ▼
java.util.Collections is a utility class with only static methods for operating on collections:
• sort(list), binarySearch(list, key)
• reverse(list), shuffle(list), rotate(list, dist)
• min(coll), max(coll), frequency(coll, o)
• unmodifiableList(list) — wrapper that throws on modification
• synchronizedList(list) — thread-safe wrapper
• nCopies(n, obj) — immutable list of n copies
• singletonList(obj) — immutable single-element list
31
What is the covariant return type in Java?
Medium ▼
Since Java 5, a subclass can override a method and return a more specific (narrower/subtype) return type. This is covariant return. Example: if parent's clone() returns Object, the child can override and return ChildClass. This avoids explicit casting at the call site. The Builder pattern uses this — each method returns this (the concrete builder type) for method chaining.
32
What happens when you put a duplicate key in HashMap?
Easy ▼
The old value for that key is replaced by the new value. put() returns the old value if the key existed, or null if it was a new key. The size of the map remains the same. Use putIfAbsent() if you want to only insert when the key doesn't exist. Use merge() if you want to combine old and new values (e.g., for frequency counting).
33
What is the difference between fail-fast and fail-safe iterators?
Hard ▼
Fail-fast iterators: Throw ConcurrentModificationException immediately when the collection is structurally modified during iteration (uses a modCount check). Examples: ArrayList, HashMap iterators. They work on the original collection — no extra memory.

Fail-safe iterators: Work on a clone/snapshot of the collection. No exception thrown on modification during iteration. Changes made during iteration may or may not be visible. Examples: CopyOnWriteArrayList, ConcurrentHashMap. Extra memory overhead from copying.
34
What is an inner class? Types of inner classes in Java?
Medium ▼
A class defined within another class. Four types:
1. Non-static inner class: Has access to outer class instance members. Requires outer instance to create.
2. Static nested class: No access to outer instance members. Can be created independently. Like a helper class scoped to the outer class.
3. Local class: Defined inside a method. Scope limited to that method.
4. Anonymous class: No name, defined and instantiated in one expression. Common for event listeners and one-off implementations before Java 8 lambdas.
35
What is the difference between Collection and Collections?
Easy ▼
Collection (interface): Root interface of the Java Collections Framework. Extended by List, Set, and Queue. Defines core methods like add(), remove(), contains(), size(), iterator().

Collections (class): A utility class with only static methods for operating on collections — sorting, searching, reversing, shuffling, creating unmodifiable wrappers, etc. Similar to how Arrays is a utility class for arrays.
36
What is a record in Java (Java 16+)?
Medium ▼
A record is a special class for holding immutable data. It auto-generates: constructor, private final fields, getters, equals(), hashCode(), and toString().

record Point(int x, int y) {} is equivalent to a full immutable class with those 5 things manually written. Records are implicitly final, cannot extend other classes, and all fields are implicitly final. Ideal for DTOs, response objects, coordinates, and value types. Significantly reduces boilerplate.
37
How is a HashMap different from a LinkedHashMap?
Easy ▼
HashMap: No guaranteed iteration order. Slightly less memory (just the hash table). Best for pure key-value lookup when order doesn't matter.

LinkedHashMap: Maintains a doubly-linked list alongside the hash table to preserve insertion order (or access order if configured). Slightly more memory and slightly slower than HashMap. Use when you need predictable iteration order or when building an LRU Cache (set accessOrder=true and override removeEldestEntry()).
38
What is method hiding vs method overriding?
Hard ▼
Method Overriding: An instance method in a subclass replaces the parent's method. Resolved at runtime via dynamic dispatch — calling through a parent reference calls the child's version.

Method Hiding: When a subclass defines a static method with the same signature as a parent static method. Resolved at compile time based on the reference type — calling through a parent reference calls the parent's static method, calling through child calls child's. This is NOT polymorphism. Use @Override to verify — it won't compile on a static method that attempts override.
39
What is the diamond problem and how does Java solve it?
Hard ▼
The diamond problem occurs when class D inherits from B and C, both of which extend A. If B and C override the same A method, D doesn't know which to use.

Java's solution: classes cannot extend multiple classes. But interfaces CAN have default methods, so a class implementing two interfaces with the same default method MUST override that method in the class itself. If an interface method and a class method conflict, the class wins. If both are interface defaults, explicit override required: InterfaceName.super.method() to choose one.
40
What is the difference between Enumeration and Iterator?
Easy ▼
Enumeration (legacy): From Java 1.0. Works on legacy classes (Vector, Hashtable). Methods: hasMoreElements(), nextElement(). Cannot remove elements during iteration. Fail-safe.

Iterator (modern): From Java 2. Works on all Collection classes. Methods: hasNext(), next(), remove(). Can safely remove elements during iteration. Fail-fast (throws CME on structural modification). Use Iterator in all modern code.
41
What are varargs in Java?
Easy ▼
Variable-length arguments (type... name): allows passing zero or more arguments of the same type. Internally treated as an array. Must be the last parameter. Only one vararg per method.

public int sum(int... nums) { int s=0; for(int n:nums) s+=n; return s; }
sum(1,2,3), sum(1), sum() all valid.

Used in: System.out.printf(format, args...), Arrays.asList(T... a), logging frameworks, and any flexible-argument utility methods.
42
What is the purpose of the Optional class?
Medium ▼
Optional<T> (Java 8) is a container that may or may not hold a non-null value. It's a better alternative to returning null, forcing the caller to explicitly handle the absent case.

Creation: Optional.of(val) (throws if null), Optional.ofNullable(val) (allows null), Optional.empty()
Usage: opt.isPresent(), opt.get(), opt.orElse(default), opt.orElseGet(supplier), opt.ifPresent(consumer), opt.map(fn), opt.filter(pred)

Best used for return types of methods that might not find a value. NOT recommended as method parameters or fields.
43
What is the difference between Iterable and Iterator?
Medium ▼
Iterable<T>: An interface indicating that a class can be iterated. Has one method: Iterator<T> iterator(). Classes implementing Iterable can be used in enhanced for-loops. The Collection interface extends Iterable.

Iterator<T>: The actual cursor/traversal mechanism. Has hasNext(), next(), remove(). Created fresh from iterable.iterator(). Stateful — tracks current position. One iterator per traversal.
44
What is a WeakHashMap?
Hard ▼
WeakHashMap uses weak references for its keys. When a key is no longer strongly referenced by any other part of the program, the GC can collect it, and the corresponding entry is automatically removed from the map. Useful for memory-sensitive caches and implementing canonical mappings.

Use case: Storing metadata about objects (like image thumbnails) where the entry should automatically disappear when the original object is garbage-collected — no manual cleanup needed. Not for general-purpose use due to unpredictable removal timing.
45
What is the difference between Runnable and Callable?
Medium ▼
Runnable: void run() — no return value, cannot throw checked exceptions. Used with Thread or ExecutorService.execute().

Callable<V>: V call() throws Exception — returns a value, can throw checked exceptions. Used with ExecutorService.submit() which returns a Future<V> to retrieve the result later.

Prefer Callable when you need the computation result or when the task might throw checked exceptions that you need to handle.
46
What is a marker interface? Examples?
Easy ▼
A marker interface has NO methods or fields — it's an empty interface that "marks" or tags a class to indicate it has a certain capability or property. The JVM or frameworks check for this using instanceof.

Examples:
• Serializable — JVM only serializes objects whose class implements this
• Cloneable — enables Object.clone() without throwing CloneNotSupportedException
• Remote (RMI) — marks object as remotely accessible

In modern Java, annotations like @Entity, @Service serve the same purpose more expressively.
47
What is type erasure in generics?
Hard ▼
At compile time, Java verifies generic type safety. At runtime, all generic type information is erased — List<String> becomes just List in bytecode. This maintains backward compatibility with pre-generic code.

Consequences:
• Cannot do new T() or new T[] inside generic code
• instanceof cannot check generic types: list instanceof List<String> won't compile
• Generic type info not available at runtime via reflection (without tricks)
• Unchecked cast warnings when mixing raw types with generics

Workaround: pass Class<T> as a token (reified type) to runtime methods.
48
What is the difference between Iterator's remove() and Collection's remove()?
Medium ▼
Collection.remove(o): Called directly on the collection during iteration. This is a structural modification that increments the modCount, causing the iterator to throw ConcurrentModificationException on the next call.

Iterator.remove(): Removes the element most recently returned by next(). This is safe during iteration because the iterator updates its own internal state along with the collection's modCount. Must call next() before each remove() call.
49
How does the LRU Cache work in Java?
Hard ▼
LRU (Least Recently Used) Cache evicts the least recently accessed entry when full.

Using LinkedHashMap (simplest): Pass accessOrder=true to maintain access order. Override removeEldestEntry() to evict when size exceeds capacity. O(1) all operations.

Manual implementation: Use a HashMap (for O(1) get) + doubly-linked list (for O(1) move-to-front and eviction). On get: find node via map, move to front. On put: if exists update and move to front; if new add to front and map; if full evict tail and remove from map.
50
Which collection would you choose for each scenario?
Hard ▼
• Frequent random access by index: ArrayList
• Frequent insert/delete at both ends: ArrayDeque
• LIFO stack: ArrayDeque (as stack)
• FIFO queue: ArrayDeque or LinkedList
• Priority-based processing (min/max): PriorityQueue
• Fast key-value lookup, order unimportant: HashMap
• Key-value with insertion order: LinkedHashMap
• Key-value with sorted keys / range queries: TreeMap
• Unique elements, fast lookup: HashSet
• Unique elements, sorted order: TreeSet
• LRU Cache: LinkedHashMap(cap, 0.75f, true)
• Thread-safe map: ConcurrentHashMap
• Immutable list: List.of()
